# Craft — PRD

**Status**: Draft
**Date**: 2026-07-09
**Author**: design brainstorming session

---

## 1. Overview

**Craft** is an AI-native game development engine. The thesis: a game engine whose architecture is designed so that AI agents (LLMs) can read, modify, and reason about game logic as first-class operations—not as a wrapper around a human-oriented engine.

The v1 deliverable is the engine **core** (scene graph, signals, resources, behavior runtime, determinism, hot reload) plus a **terminal-rendered reference game** (snake) that demonstrates an agent building a complete, playable, replayable, hot-reloadable game through the public API.

**Reference for design**: Godot (scene tree, signals, resources). Craft borrows design ideas, not code. The engine is implemented from scratch in Rust, with a TypeScript SDK as the primary agent-facing surface.

---

## 2. Vision & Thesis

### 2.1 Problem

Modern game engines (Unity, Unreal, Godot) target human developers. Their primary authoring representations—C# scripts, C++ classes, GDScript bytecode, scene graph XML—were designed for human readability and execution performance, not for LLM consumption. When an AI agent operates on these engines, the bottleneck is representation:

- LLM-written scripts are unreliable (syntax errors, scope mistakes, framework misuse)
- Scene graphs are deep-nested and use binary or hard-to-parse formats
- Runtime state is opaque (no structured inspection)
- Iteration is slow (edit → compile → run → restart)
- Testing requires human-graded play sessions

### 2.2 Thesis

A game engine designed for AI agents has different core properties:

| Property | Why it matters for agents |
|---|---|
| **Game logic = structured data** (JSON, not bytecode) | LLM writes and edits data reliably; schemas validate every change |
| **Closed-set action vocabulary** | Every action the agent can take is enumerable and verifiable |
| **Inspectable runtime state** | Agent can query the entire game tree at any tick |
| **Deterministic execution** | Replay, regression tests, and counterfactual reasoning all work |
| **Hot reload as default** | Iteration loop is "edit JSON → see effect in <100ms" |
| **Schema is first-class** | Engine publishes its own API surface as JSON Schema; SDK types auto-generate; agent never sees drift |
| **No GUI editor** | The agent is the editor. Skipping a GUI eliminates a huge category of unrewarding work |

### 2.3 What this is NOT

- Not an "AI wrapper around Godot." We reimplement.
- Not a no-code tool. The agent writes JSON; the engine interprets it.
- Not a runtime AI/NPC system (v1). That is v2+.
- Not a generative-asset pipeline (v1). ASCII only.
- Not a competitor to Unity/Unreal at the AAA level. We're building for a different audience.

## 3. Goals & Non-Goals

### 3.1 v1 Goals

1. **Engine core is correct and complete enough to run a real game**: scene graph + signals + resources + behavior runtime + determinism + hot reload.
2. **An AI agent can build the reference game (snake) end-to-end** using only the public TypeScript SDK.
3. **The reference game is replayable**: same scene + same input + same seed = byte-identical state.
4. **The reference game supports hot reload**: editing scene.json during a run applies changes without restart.
5. **The SDK is schema-driven**: every Rust type exports JSON Schema; TS types are auto-generated.
6. **Errors are first-class structured data**, with location, context, and actionable suggestions.
7. **An agent evaluation harness exists** with at least 4 benchmark tasks.

### 3.2 v1 Non-Goals (Explicitly Deferred)

- 2D / 3D rendering (proper graphics, sprites, models)
- Physics (collision detection beyond AABB/circle, rigid bodies, joints)
- Asset import pipeline (images, audio, fonts — v1 is ASCII only)
- Multiplayer, networking, replication
- GUI editor (the agent is the editor; no ImGui, no Electron app)
- Mobile / console targets
- Plugin system / sandboxed agent execution
- Visual scripting (rejected in favor of typed JSON)
- Lua / Python / GDScript-style embedded scripting (Rust DSL only as escape hatch for systems)

### 3.3 Success Criteria (v1 done means)

- All M1–M10 milestones pass their testable acceptance criteria (§12)
- Reference game (snake) builds, runs for 1,000 ticks, replays to hash-equal state, hot-reloads at tick 500 without abort (§10.3)
- AI agent (defined in §10.2: LLM with pinned model/prompt/seed) completes at least 3/4 benchmark tasks in <30 minutes each, **reproducibly** (byte-identical pass/fail across two runs)
- Performance budgets in §10.0 met on the reference machine
- Test coverage on engine core ≥80%
- All `cargo test`, `npm test`, replay regression tests, perf tests green
- Documentation: PRD (this doc), ARCHITECTURE.md, AGENT_GUIDE.md, SCHEMA.md all non-stub

## 5. Architecture

### 5.1 System Diagram

```
┌────────────────────────────────────────────────────┐
│  Agent (LLM, e.g. Claude / Codex)                  │
│  - Writes scene JSON, behavior rules, resource specs│
│  - Queries runtime state, runs scenarios, replays  │
└──────────────────────┬─────────────────────────────┘
                       │ TypeScript SDK (typed client)
┌──────────────────────▼─────────────────────────────┐
│  JSON-RPC bridge (Rust, in-process)                │
│  - Typed API surface                               │
│  - Rust types → JSON Schema                        │
│  - NAPI binding for TS                             │
└──────────────────────┬─────────────────────────────┘
┌──────────────────────▼─────────────────────────────┐
│  Engine kernel (Rust crates)                       │
│  ├─ craft-kernel: scene graph, signals, resources, │
│  │   behavior runtime, hot reload                  │
│  ├─ craft-replay: seeds, tick, replay, recording   │
│  ├─ craft-schema: Rust types → JSON Schema         │
│  └─ craft-bridge: JSON-RPC + NAPI binding          │
└──────────────────────┬─────────────────────────────┘
┌──────────────────────▼─────────────────────────────┐
│  Reference game (v1)                                │
│  ├─ craft-terminal: ANSI terminal renderer (crate) │
│  └─ games/snake/: scene.json + behavior rules      │
└────────────────────────────────────────────────────┘
```

### 5.2 Key Architectural Decisions

1. **Same process; RPC is API, not transport**: JSON-RPC runs in-process via **sync NAPI**, with `Promise.resolve` wrappers. No network overhead in the v1 loop, no thread-safety concerns, no async-during-tick reentrancy. Sidecar mode deferred.
2. **Schema is a first-class product**: every Rust type — including **every action verb** — emits JSON Schema. TS SDK types are generated. The agent's API view cannot drift from engine reality.
3. **Game content = pure data**: scenes, behaviors, resources are all JSON. The engine interprets; it does not compile. **This is the core of "AI-native."**
4. **Behavior is data, with a Rust escape hatch**: v1 ships three behavior primitives (state machine, tick rule, signal handler) that compose a closed set of actions. When the closed set is insufficient, agents call into Rust-defined `system` functions through a typed JSON contract (registered via `craft_system!` macro into a `craft-kernel` registry).

### 5.3 Engine Kernel Module Boundaries

| Module | Responsibility | Does NOT |
|---|---|---|
| `scene-graph` (in `craft-kernel`) | Node hierarchy, typed node properties | Interpret behavior |
| `signals` (in `craft-kernel`) | Event publish/subscribe, typed payloads (including the reserved `tick` signal) | Implement state machines |
| `resources` (in `craft-kernel`) | Typed data containers, addressable, immutable | Cache strategy |
| `behavior` (in `craft-kernel`) | Interpret declarative rules, schedule systems, expose action vocabulary | Render |
| `systems` (in `craft-kernel`) | Rust-side registry for `craft_system!`-registered functions, schema-exported | Game logic itself |
| `replay` (in `craft-replay`) | Seeds, tick control, **recording codec**, replay | Behavior authoring |
| `hot-reload` (in `craft-kernel`) | File watch, state diff, hot-patch | Version control |
| `lint` (in `craft-kernel`) | Static scene analysis (signals-with-no-subscribers, unreachable states, unused components, undefined references, broken resource paths) | Runtime behavior |
| `renderer` (in `craft-terminal`) | Terminal (ANSI) rendering only | Any other rendering backend |

`craft-replay` is a separate crate from `craft-kernel` because the recording codec has its own storage/serialization concerns. All kernel-internal modules are one-way: they communicate through traits, not direct state import.

## 6. Core Abstractions

### 6.1 Scene Graph

A **scene file** is a top-level JSON object with a `kind: "scene"` discriminator. The engine uses this discriminator (plus `res://` path conventions) to distinguish scenes from resources and behavior files.

```json
{
  "kind": "scene",
  "id": "snake_main",
  "root": { /* the root node */ }
}
```

A **node** is a typed entity with parent/child relationships, components, and behaviors. All nodes are expressed as JSON.

```json
{
  "id": "player",
  "type": "Player",
  "parent": "world",
  "components": {
    "position": [10, 10],
    "velocity": [0, 0],
    "direction": "right",
    "score": 0
  },
  "behaviors": [ /* see §6.4 */ ]
}
```

Node types have two layers:
- **Base `Node`**: empty container, hierarchy only. Always available.
- **Typed nodes** (user-defined, e.g. `Player`): defined in Rust via the `craft_node!` macro, which:
  1. Declares the type's required components and their types.
  2. Auto-registers the type into the engine's type registry.
  3. Auto-generates a JSON Schema fragment for the type.
  4. Surfaces the type through `engine.getSchema()` for agent introspection.

Schemas validate at load time; agents get structured errors (§9) on violation.

**Built-in typed node**: `Input` — a singleton that exposes the current input frame as components (`input.direction`, `input.action`, etc.). The engine's input layer populates this each tick. Agent scenes typically include one `Input` node at the root.

### 6.2 Signals

A typed event bus for cross-node communication. Subscriptions can be:
- **Internal**: declared as part of a node's behaviors.
- **External**: agent subscribes via RPC for observation.

**Reserved signals** (fired by the engine kernel, not user code):
- `tick` — fired exactly once per scheduler tick, before any `on_tick` rule fires. State machine transitions with `event: "tick"` listen to this.
- `hot_reload` — fired after a successful hot-reload; payload includes the diff.
- `replay_start` / `replay_end` — fired at replay boundaries.

User-defined signals coexist with these; the engine never reserves user signal names.

```json
// Emit (as an action, see §6.4)
{ "kind": "emit", "signal": "eaten", "args": { "by": "enemy_1" } }

// Node-internal subscription
{ "kind": "on_signal", "signal": "eaten", "actions": [...] }

// Agent subscription (via RPC)
engine.subscribe("eaten", (event) => { ... });
```

### 6.3 Resources

Strongly-typed, immutable, reusable across scenes. Addressed by `res://path`.

```json
{
  "resource": "res://foods/apple.json",
  "type": "Food",
  "data": { "color": "red", "points": 10, "sprite": "res://sprites/apple.asc" }
}
```

Resources are **immutable**: runtime modification does not affect already-loaded instances. This is critical to prevent agent-induced spooky-action bugs. For mutable data, use components (per-node).

### 6.4 Behaviors

The agent's primary authoring surface. v1 ships three primitives plus one escape hatch. **Every action is a structured JSON object with a fixed shape** — no string DSL, no colon-separated verbs, no implicit parsing. This is the AI-native choice: actions are schema-validated data, not parseable text.

#### (a) State Machine (per node)

```json
{
  "kind": "state_machine",
  "states": ["idle", "moving", "dead"],
  "initial": "idle",
  "transitions": [
    { "from": "idle", "event": "tick", "to": "moving", "guard": { "neq": ["input", "none"] } },
    { "from": "moving", "event": "tick", "to": "idle", "guard": { "eq": ["input", "none"] } },
    { "from": "*", "event": "collide", "to": "dead" }
  ]
}
```

`event` is a signal name (e.g. `"tick"` is the reserved tick signal from §6.2). `guard` is a structured expression tree (see §6.5 for the expression vocabulary).

#### (b) Tick Rule (per node)

```json
{
  "kind": "on_tick",
  "actions": [
    { "kind": "move", "target": "self", "by": "input.direction" },
    { "kind": "if", "cond": { "eq": ["self.position", "food.position"] },
      "then": [
        { "kind": "set_state", "target": "self", "key": "length", "value": { "add": ["self.length", 1] } },
        { "kind": "emit", "signal": "eaten" }
      ]
    }
  ]
}
```

#### (c) Signal Handler (per node)

```json
{
  "kind": "on_signal",
  "signal": "eaten",
  "actions": [
    { "kind": "set_state", "target": "player", "key": "dead", "value": true },
    { "kind": "emit", "signal": "game_over" },
    { "kind": "destroy", "target": "self" }
  ]
}
```

#### (d) Escape Hatch: System Call

When the closed action set is insufficient (pathfinding, math, external algorithms), agents invoke a Rust-defined system:

```json
{
  "kind": "call_system",
  "system": "pathfinding::astar",
  "args": { "from": "self.position", "to": "target.position" },
  "result_in": { "key": "next_path", "on": "self" }
}
```

Systems are strongly-typed Rust functions registered via the `craft_system!` macro into the `craft-kernel` system registry. The registry:
- Surfaces registered systems through `engine.listSystems()`.
- Auto-generates JSON Schema for each system's signature.
- Pins return types (single value, struct, or list of structs) so `result_in` is type-checked at call time.

**Why structured JSON, not a string DSL**: a colon-separated DSL (`move:player:input.direction`) is the worst representation for an LLM — it embeds parsing ambiguity, requires a grammar, and defeats schema validation. Structured JSON actions are unambiguous, schema-validatable per-verb, and trivially diff-able for hot reload.

### 6.5 Action Vocabulary (closed set, structured)

Every action is a JSON object with `kind: <verb>` and a verb-specific payload. The engine publishes the schema for each verb via `engine.getActionSchema(verb)` (see §8.1). The vocabulary is **closed** — agents cannot invent new verbs; the engine never "half-executes" an action.

v1 verbs:

| `kind` | Payload shape | Effect |
|---|---|---|
| `set_state` | `{ target, key, value }` | Write a component on the target node |
| `get_state` | `{ target, key, into }` | Read a component into a local var |
| `emit` | `{ signal, args }` | Fire a signal (queued; resolved next tick) |
| `destroy` | `{ target }` | Remove a node (errors if other nodes reference it) |
| `spawn` | `{ type, parent, components }` | Create a new node at runtime |
| `if` | `{ cond, then, else? }` | Conditional execution (used inside `actions: [...]`) |
| `move` | `{ target, by }` | Apply a delta to a position component |
| `animate` | `{ target, key, to, duration }` | Interpolate a component over ticks |
| `log` | `{ level, message, fields }` | Debug output (also surfaces in agent subscription stream) |
| `call_system` | `{ system, args, result_in }` | Invoke a registered Rust system |

**Expression vocabulary** (for `cond`, `guard`, `value`, `by`, `args`): a closed set of structured expressions. **No free-form expressions.** Agents compose from the vocabulary; the engine validates type-correctness at parse time.

- **References**: `{ "ref": "nodeId.componentKey" }`. Bare strings in expression position are shorthand for `{ "ref": <string> }` (e.g. `"self.position"` ≡ `{ "ref": "self.position" }`). Reserved tokens: `self` (current node), `none` (null), `true` / `false`, integer/decimal literals.
- **Comparisons**: `{ "eq": [a, b] }`, `{ "neq": [a, b] }`, `{ "lt": [a, b] }`, `{ "gt": [a, b] }`.
- **Arithmetic**: `{ "add": [a, b] }`, `{ "sub": [a, b] }`.

`set_state` with an expression `value` evaluates the expression against the current state, then writes the result. Read-then-write of the same component (e.g. `{ "add": ["self.score", 1] }` incrementing `self.score`) is supported.

**Discoverability**: `engine.getSchema()` returns:
- All node types
- All action verbs + their schemas
- All systems + their signatures
- All signal names (built-in + user-defined)
- All expression operators

The agent can call `engine.getActionSchema("emit")` to see exactly what `emit` accepts, in JSON Schema. No hardcoded knowledge required.

## 7. Data Flow

### 7.1 Three Core Loops

**Authoring loop** (agent writes a scene):
```
agent writes scene.json
   ↓
Engine: parse → schema validate
   ↓
Fail? → structured error (file, line, JSONPath, suggestion)
Pass? → load into scene graph
```

**Tick loop** (every frame, default 60Hz):
```
1. InputBus populates the Input node's components
2. Tick scheduler fires the reserved `tick` signal
3. For each node in scene-graph declaration order:
   a. State machine: evaluate transitions triggered by `tick` or any queued signal
   b. `on_tick` rules: actions execute in the order they appear in the JSON
   c. `on_signal` handlers for signals queued before this tick: execute in
      node declaration order
4. Updates from this tick are batched and committed
5. Signals emitted during this tick are queued for the next tick
6. Renderer reads committed components
```

**Tick ordering rules** (for replay determinism):
- **Node order**: declaration order in the scene JSON (stable across reloads).
- **Within a node**: state machine fires first, then `on_tick` rules, then `on_signal` handlers (in `signal_name` lexicographic order, then by subscription registration order).
- **Within an action list**: declaration order in the JSON. Subsequent actions see prior actions' effects within the same tick.
- **Across ticks**: signals emitted this tick do not fire handlers this tick. A tick is atomic from any action's perspective.
- **`call_system` results are visible to subsequent actions in the same tick** (systems are sync).

**Observation loop** (agent watches the game):
- Subscribe to signal: real-time event stream
- Query state: snapshot of current components
- Both via JSON-RPC; TS SDK exposes `engine.subscribe` and `engine.getState`.

### 7.2 Hot Reload

```
file watcher detects scene.json change
   ↓
re-parse + validate
   ↓
diff old vs new scene graph:
  - node property change → update component
  - new node → spawn
  - removed node → destroy (referenced = error & rollback)
  - behavior change → replace
   ↓
no breaking change → seamless continuation
```

**Invariant**: node IDs are stable. Hot reload preserves them. Agent subscriptions survive.

**Resource hot-reload semantics** (committed for v1):
- `engine.registerResource(path, data)` writes new content under `path`.
- **Already-loaded instances keep the old copy** until they are unloaded and re-fetched.
- Newly spawned nodes that reference the path see the new content.
- This preserves the idempotency of hot reload: an agent that re-registers a resource mid-game cannot retroactively change the meaning of already-spawned entities.
- Recordings (replay §7.3) embed resource content snapshots, not just paths, so replays are independent of subsequent `registerResource` calls.

### 7.3 Replay

Recording = `scene.json (frozen copy) + seed + input_log + tick_count + per-tick state hash + resource snapshots`. Replay = re-run with same seed, same input. Any tick can be inspected, paused, stepped. Agents use this for **counterfactual reasoning**: "what would have happened if state machine X had a different transition?"

Replays embed resource content snapshots so they remain reproducible even after subsequent `registerResource` calls in the same project.

### 7.4 Invariants

| Invariant | Reason |
|---|---|
| Tick is atomic: an action cannot observe its own emit's effect | Prevent reentrancy; enable replay |
| Tick action ordering is deterministic and documented (§7.1) | Replay hash equality is testable |
| Node A cannot directly `set_state` node B's component | Force explicit dependencies via signals |
| Hot reload preserves IDs | Agent subscriptions survive |
| Hot reload of resources does not retroactively change already-loaded instances | Idempotency; agent cannot corrupt state via re-register |
| System calls are sync-blocking (via NAPI sync mode) | Async systems would break replay and tick atomicity |
| Closed-set actions cannot perform external IO | Behaviors stay replayable and inspectable |
| Engine is single-threaded for v1 | Determinism; no data races; agents reason about one timeline |
| Renderer is read-only (reads components, never writes) | Render output cannot influence game state |
| Recordings persist their tick rate; replays run at the recorded rate | Replay determinism is independent of current `setTickRate` |

## 8. API Surface (TypeScript SDK)

The SDK is the agent's window into the engine. **Schema is the source of truth**: every Rust type emits JSON Schema; TS types are generated. The agent's view of the API cannot drift from the engine.

### 8.1 Method Surface

```typescript
// ── Engine lifecycle ──
engine.start(scenePath: string): Promise<SceneId>
engine.stop(): Promise<void>
engine.reload(): Promise<void>          // trigger hot reload
engine.setTickRate(hz: number): Promise<void>   // default 60; `setTickRate(0)` is sugar for `engine.pause()`

// ── Scene / nodes ──
engine.listNodes(): Promise<NodeInfo[]>
engine.getNode(id: NodeId): Promise<NodeState>
engine.getSceneGraph(): Promise<SceneGraph>
engine.validateScene(scene: Scene): Promise<ValidationResult>
engine.lint(scene: Scene): Promise<LintReport>   // static analysis; see §8.4

// ── Components ──
engine.getComponent<T>(nodeId: NodeId, key: string): Promise<T>
engine.setComponent(nodeId: NodeId, key: string, value: any): Promise<void>

// ── Signals ──
engine.subscribe(signal: string, handler: (event) => void): Unsubscribe
engine.emit(signal: string, args: any): Promise<void>

// ── Resources ──
engine.listResources(): Promise<ResourceInfo[]>
engine.getResource(path: string): Promise<Resource>
engine.registerResource(path: string, data: any): Promise<void>   // does not affect already-loaded instances

// ── Systems ──
engine.listSystems(): Promise<SystemInfo[]>                       // name + signature + return type
engine.getSystemSchema(system: string): Promise<JSONSchema>

// ── Replay / debugging ──
engine.startRecording(path: string): Promise<void>
engine.replay(recording: Recording, opts?: { fromTick?: number }): Promise<void>
engine.step(): Promise<void>     // single tick
engine.pause(): Promise<void>
engine.resume(): Promise<void>
engine.getStateAtTick(tick: number): Promise<Snapshot>
engine.diff(tickA: number, tickB: number): Promise<SnapshotDiff>   // see §8.4

// ── Schema introspection ──
engine.getSchema(): Promise<JSONSchema>            // all node types, actions, systems
engine.getNodeTypeSchema(type: string): Promise<JSONSchema>
engine.getActionSchema(verb: string): Promise<JSONSchema>   // per-verb payload schema

// ── AI-native primitives ──
engine.dryRun(nodeId: NodeId, actions: Action[]): Promise<ComponentDiff>   // see §8.4
engine.explain(nodeId: NodeId): Promise<NodeExplanation>                   // see §8.4
```

**Concurrency model**: the SDK uses **sync NAPI bindings** wrapped in `Promise.resolve`. All engine calls are O(1) sync operations on the same thread as the agent. This preserves tick atomicity and determinism — the engine never has multiple in-flight requests. The agent's event loop is unaffected because all RPCs are immediate.

### 8.2 Typical Agent Round Trip

```typescript
await engine.start("res://games/snake/scene.json");

// 1) Self-describe
const actions = await engine.getSchema();
// → { actions: { emit: { ... }, set_state: { ... }, ... }, nodeTypes: {...}, systems: {...} }

// 2) Observe
const playerPos = await engine.getComponent("player", "position");
engine.subscribe("eaten", e => console.log("score:", e.score));

// 3) Hypothetical (no side effects)
const diff = await engine.dryRun("player", [
  { kind: "move", target: "self", by: "input.direction" }
]);
// → { components: { position: [11, 10] } } — predicted, not applied

// 4) Modify (write to file, trigger hot reload)
await fs.writeFile(scenePath, updatedScene);
await engine.reload();

// 5) Counterfactual reasoning
await engine.replay(recording, { fromTick: 42 });
```

### 8.3 Self-Description

`engine.getSchema()` returns the entire engine schema — every node type, every action verb, every system, every signal name. The agent queries this on startup. **No hardcoded knowledge of the engine is required.** This is critical: future engine versions require no agent retraining.

### 8.4 AI-Native Primitives

These four methods make the engine meaningfully more agent-native — the difference between "valid JSON config" and "AI-native by construction."

**`engine.lint(scene)`** — static analysis of a scene before load. Detects:
- Signals with no subscribers
- State machines with unreachable states
- Components declared but never read
- Node IDs referenced but undefined
- Behaviors whose actions reference missing targets/systems
- Resource paths that don't resolve

Returns a `LintReport` (same shape as §9.2 errors, but with `severity: "warning"` by default). Cheap to implement (a few hundred LOC of static analysis), high agent value.

**`engine.dryRun(nodeId, actions)`** — execute a proposed action list against the current state and return the resulting component diff, **without** actually mutating. Lets the agent ask "what would happen if I emit `eaten` right now?" without side effects. Useful for replanning after errors. **Signals emitted during a dry run are discarded** (not queued, not delivered). Systems called during a dry run must be pure (no engine-state mutation outside the dry-run buffer); impure systems are rejected with a `dryRun:system_must_be_pure` error.

**`engine.explain(nodeId)`** — return a `NodeExplanation`: a structured object with `summary` (one-line natural-language description, e.g. `"Player at (10, 10), moving right, score 50, state: playing, 3 children"`), `components` (current values, typed), `active_state` (current state-machine state if any), and `signal_subscriptions` (the names of signals this node listens to). The agent can include this verbatim in its context window. This format is what LLMs reason about best. The exact shape is exposed via `engine.getNodeExplanationSchema()`.

**`engine.diff(tickA, tickB)`** — return a `SnapshotDiff`: a structured object listing component changes (per node, per key: `{ nodeId, key, before, after }`), node creations, node destructions, and signal emissions between the two ticks. The agent uses this for "what changed since I last looked?" debugging. The exact shape is exposed via `engine.getSnapshotDiffSchema()`.

## 9. Error Handling

### 9.1 Core Principle

**Errors are first-class structured data.** Agents can reason about them programmatically; the format is consistent across all engine subsystems.

### 9.2 Error Format

```json
{
  "category": "validation",
  "code": "E_SCHEMA_MISMATCH",
  "message": "Node 'player' missing required component 'position'",
  "location": { "file": "scene.json", "line": 42, "path": "$.root.children[0]" },
  "context": { "node_id": "player", "node_type": "Player", "missing_field": "position" },
  "suggestion": "Add 'position' component to node 'player'",
  "severity": "error"
}
```

| Field | Purpose |
|---|---|
| `category` + `code` | Programmatic classification (retry / ignore / abort) |
| `location` | Jump to file position (file + JSONPath) |
| `context` | Surrounding state for diagnosis |
| `suggestion` | Entry point for agent's auto-patch |
| `severity` | `error` / `warning` / `info` |

### 9.3 Error Categories

| Category | When | Example |
|---|---|---|
| `validation` | Load / validate scene | Schema mismatch, ID collision, null reference |
| `hot_reload` | File change diff | Breaking change, rollback failure |
| `runtime` | During tick | Reference to missing node, guard never true |
| `system` | Rust system call | Business error (see system's schema) |
| `replay` | During replay | Tick state divergence (fatal) |

### 9.4 Invariants

- **Engine never panics.** Every panic is converted to a structured error.
- **Validation runs once at load.** Schema errors do not surface during ticks.
- **Runtime errors are warnings, not fatal.** A single action's failure does not abort the tick (unless the system marks it fatal).
- **Replay divergence is fatal.** State mismatch indicates a bug; stop immediately.
- **Suggestions are schema-derived**, not LLM-fabricated. They are structured, executable patch hints.

## 10. Testing Strategy

### 10.0 Performance Budgets (v1 targets)

The following budgets drive the data-structure choices for the behavior runtime. They are validated by the reference game and by `bench/`-harnessed micro-benchmarks. Targets:

| Metric | Target |
|---|---|
| Active nodes per scene | 1,000 |
| Signals fired per tick | 100 |
| Actions executed per tick | 10,000 |
| Tick rate (default) | 60 Hz |
| Tick wall-clock budget (snake scene) | ≤ 8 ms |
| Hot-reload latency (snake scene) | ≤ 100 ms |
| Replay of 1,000 ticks | ≤ 5 s wall clock |
| Cold start (binary to first tick) | ≤ 500 ms |
| `engine.getSchema()` latency | ≤ 50 ms (cached after first call) |

These are committed v1 targets, not aspirations. A milestone is "done" only if it meets its perf criterion.

### 10.1 Engine Kernel

- **Unit tests**: per-module (scene graph, signals, resources, behavior, replay, hot-reload).
- **Property tests** (`proptest`): "any scene loads + serializes → byte-equal"; "any schema change → JSON Schema remains valid"; "tick action ordering is stable across runs"; "any scene + any input → replay hash equals original".
- **Replay regression**: reference game recordings (1,000+ ticks) checked in. Every change re-runs replay + hash check.
- **Perf tests**: micro-benchmarks in `craft-kernel/benches/` for the 10,000-actions/tick target. CI fails on regression >10%.

### 10.2 AI Agent Evaluation (Core, Novel)

**Agent definition** (commit before M10, and required for any benchmark):

- **Agent** = an LLM with pinned `(model, prompt template, seed, tool-use schema)`.
- **Model**: v1 pins a specific model identifier (e.g. `claude-sonnet-4-5` or `gpt-5` family — finalized at M10). Pinning means: the same identifier is used in every evaluation run.
- **Prompt template**: a versioned template in `benchmarks/prompts/` that includes task description, schema, and a small set of worked examples.
- **Tool-use schema**: the engine's `engine.*` methods exposed to the LLM in OpenAI / Anthropic tool-use format. The schema is auto-generated from `engine.getSchema()`.
- **Seed**: any sampling seed the model supports. Pinned.

**Eval harness** (CLI tool, in `benchmarks/runner`):

```
cargo run --bin bench -- \
  --model claude-sonnet-4-5 \
  --prompt benchmarks/prompts/snake.md \
  --task benchmarks/tasks/snake.toml \
  --output benchmarks/results/snake-2026-07-09.json
```

The runner: (1) starts the engine, (2) feeds the task + tool schema to the LLM, (3) records the agent's session (every engine call), (4) replays the session deterministically, (5) computes pass/fail against the task's oracle.

**Three evaluation layers**:

| Layer | Question | Method |
|---|---|---|
| Compile | Does the agent's scene load? | Engine `validateScene` + `lint` pass rate |
| Behavior | Does the game run? | Recording + replay + hash equality (no divergence) |
| Task | Does the agent complete the task? | Task-specific oracle: win condition, state assertion, or scripted play-through |

**Reproducibility criterion** (commit before M10): given fixed `(model, prompt, seed, task)`, two eval runs produce **byte-identical pass/fail outcomes** and **identical recorded sessions** (modulo LLM-side non-determinism, which is bounded by setting `temperature: 0` and pinning the model identifier).

**Benchmark task suite** (v1, ≥4 tasks): snake, a small puzzle, a tower-defense fragment, a dialog tree. Each task has a `task.toml` declaring: the goal, the win condition, time budget (≤30 min agent wall clock), and the scoring oracle.

### 10.3 Reference Game's Dual Role

Not just a demo — also the integration test. Any engine change must keep the reference game working:
- **Acceptance**: snake game loads, runs for 1,000 ticks without error, replay produces hash-equal state, hot-reload of the snake scene at tick 500 does not abort the run. All checked in CI.

### 10.4 Testable Invariants

| Invariant | Test |
|---|---|
| Tick atomicity | Unit: emit in same tick does not change state |
| Tick ordering is deterministic | Property: 100 random scenes × 100 random input sequences produce stable action order |
| Hot reload preserves IDs | Unit: subscription still fires after reload |
| Replay reproducibility | Integration: recording + replay hash equal across 10 reruns |
| Schema roundtrip | Property: scene JSON byte-equal after parse/serialize |
| Closed actions don't perform IO | Static analysis on every action + runtime check on `call_system` |
| `dryRun` does not mutate | Unit: state hash unchanged after dryRun |
| Lint catches all references to undefined symbols | Property: any scene with a missing reference produces a lint warning |

### 10.5 Acceptance Criteria Tightening (M3, M9, M10)

Replacing vague acceptance with testable criteria:

- **M3** — "all runnable" becomes: each of the 10 verbs in §6.5 has at least one test scene that exercises the verb and asserts a known post-condition. `engine.lint` exists and catches all six classes of issue in §8.4.
- **M9** — "playable + replayable + hot-reloadable" becomes: snake runs for 1,000 ticks without error, replay produces hash-equal state, hot-reload at tick 500 does not abort the run.
- **M10** — "results reproducible" becomes: given fixed `(model, prompt, seed)`, the eval script produces identical pass/fail outcomes across two runs (LLM temperature 0).

## 14. Glossary

- **Action** — a structured JSON object with `kind: <verb>` and verb-specific payload. Closed vocabulary.
- **Agent** — an LLM with pinned `(model, prompt, seed, tool-use schema)` that uses the Craft SDK to author or modify game content.
- **Behavior** — a declarative rule attached to a node (state machine / tick rule / signal handler).
- **Component** — typed, per-node mutable data. The agent reads/writes these via `set_state` / `get_state`.
- **Craft.toml** — the project manifest declaring entry scene, default seed, tick rate, and `res_root`.
- **Dry run** — `engine.dryRun(nodeId, actions)`; executes actions hypothetically and returns the diff without mutating state. Emitted signals are discarded.
- **Explain** — `engine.explain(nodeId)`; returns a `NodeExplanation` with summary, components, active state, and subscriptions.
- **Expression** — a closed-set structured value (eq, neq, lt, gt, add, sub, ref, or a bare-string shorthand for `ref`). Used in `cond`, `guard`, `value`, `by`, `args`.
- **Lint** — `engine.lint(scene)`; static analysis of a scene returning warnings about unreachable states, undefined references, etc.
- **Node type** — a typed Rust struct registered via the `craft_node!` macro; defines the required components and emits a JSON Schema fragment.
- **Replay** — deterministic re-execution of a recorded (scene + seed + input + per-tick hash + resource snapshots) sequence. Recordings embed their tick rate.
- **Resource** — typed, immutable, reusable data. The agent creates these via `registerResource`; already-loaded instances are not retroactively affected.
- **Scene graph** — the tree of nodes that constitutes a game's object hierarchy.
- **Schema** — JSON Schema fragment emitted by a Rust type (node type, action verb, system). Surfaced via `engine.getSchema()`, `engine.getActionSchema(verb)`, etc. The agent's view of the engine.
- **Signal** — a typed event published on the engine's event bus. Reserved signals include `tick`, `hot_reload`, `replay_start`, `replay_end`.
- **Snapshot** — a point-in-time dump of the full scene-graph state (node ids, components, active states). `engine.getStateAtTick(tick)` returns one.
- **Snapshot diff** — structured delta between two snapshots; returned by `engine.diff(tickA, tickB)`.
- **System** — a Rust function with a typed JSON contract, registered via `craft_system!` and callable from behavior. The escape hatch.
- **Tick** — one unit of game simulation. Atomic from the agent's perspective. Default 60Hz.
- **Tick ordering** — declaration order in scene JSON for nodes; declaration order in actions JSON for action lists; lexicographic then registration order for signal handlers.
- **Tick rate** — Hz of the simulation; default 60. Set via `engine.setTickRate(hz)`. `setTickRate(0)` is sugar for `pause()`. Recordings persist their tick rate.
