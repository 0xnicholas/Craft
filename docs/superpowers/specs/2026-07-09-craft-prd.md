# Craft — PRD

**Status**: Draft
**Date**: 2026-07-09
**Author**: design brainstorming session

---

## 1. Overview

**Craft** is an AI-native game development engine. The thesis: a game engine whose architecture is designed so that AI agents (LLMs) can read, modify, and reason about game logic as first-class operations—not as a wrapper around a human-oriented engine.

The v1 deliverable is the engine **core** (scene graph, signals, resources, behavior runtime, determinism, hot reload) plus a **terminal-rendered reference game** (snake) that demonstrates an agent building a complete, playable, replayable, hot-reloadable game through the public API.

**Reference for design**: Godot (scene tree, signals, resources). Craft borrows design ideas, not code. The engine is implemented from scratch in Rust, with a TypeScript SDK as the primary agent-facing surface.

---

## 2. Vision & Thesis

### 2.1 Problem

Modern game engines (Unity, Unreal, Godot) target human developers. Their primary authoring representations—C# scripts, C++ classes, GDScript bytecode, scene graph XML—were designed for human readability and execution performance, not for LLM consumption. When an AI agent operates on these engines, the bottleneck is representation:

- LLM-written scripts are unreliable (syntax errors, scope mistakes, framework misuse)
- Scene graphs are deep-nested and use binary or hard-to-parse formats
- Runtime state is opaque (no structured inspection)
- Iteration is slow (edit → compile → run → restart)
- Testing requires human-graded play sessions

### 2.2 Thesis

A game engine designed for AI agents has different core properties:

| Property | Why it matters for agents |
|---|---|
| **Game logic = structured data** (JSON, not bytecode) | LLM writes and edits data reliably; schemas validate every change |
| **Closed-set action vocabulary** | Every action the agent can take is enumerable and verifiable |
| **Inspectable runtime state** | Agent can query the entire game tree at any tick |
| **Deterministic execution** | Replay, regression tests, and counterfactual reasoning all work |
| **Hot reload as default** | Iteration loop is "edit JSON → see effect in <100ms" |
| **Schema is first-class** | Engine publishes its own API surface as JSON Schema; SDK types auto-generate; agent never sees drift |
| **No GUI editor** | The agent is the editor. Skipping a GUI eliminates a huge category of unrewarding work |

### 2.3 What this is NOT

- Not an "AI wrapper around Godot." We reimplement.
- Not a no-code tool. The agent writes JSON; the engine interprets it.
- Not a runtime AI/NPC system (v1). That is v2+.
- Not a generative-asset pipeline (v1). ASCII only.
- Not a competitor to Unity/Unreal at the AAA level. We're building for a different audience.

## 3. Goals & Non-Goals

### 3.1 v1 Goals

1. **Engine core is correct and complete enough to run a real game**: scene graph + signals + resources + behavior runtime + determinism + hot reload.
2. **An AI agent can build the reference game (snake) end-to-end** using only the public TypeScript SDK.
3. **The reference game is replayable**: same scene + same input + same seed = byte-identical state.
4. **The reference game supports hot reload**: editing scene.json during a run applies changes without restart.
5. **The SDK is schema-driven**: every Rust type exports JSON Schema; TS types are auto-generated.
6. **Errors are first-class structured data**, with location, context, and actionable suggestions.
7. **An agent evaluation harness exists** with at least 4 benchmark tasks.

### 3.2 v1 Non-Goals (Explicitly Deferred)

- 2D / 3D rendering (proper graphics, sprites, models)
- Physics (collision detection beyond AABB/circle, rigid bodies, joints)
- Asset import pipeline (images, audio, fonts — v1 is ASCII only)
- Multiplayer, networking, replication
- GUI editor (the agent is the editor; no ImGui, no Electron app)
- Mobile / console targets
- Plugin system / sandboxed agent execution
- Visual scripting (rejected in favor of typed JSON)
- Lua / Python / GDScript-style embedded scripting (Rust DSL only as escape hatch for systems)

### 3.3 Success Criteria (v1 done means)

- All M1–M10 milestones pass
- Reference game (snake) builds, runs, replays, hot-reloads
- AI agent can complete at least 3/4 benchmark tasks in <30 minutes each
- Test coverage on engine core ≥80%
- All `cargo test`, `npm test`, replay regression tests green
- Documentation: PRD (this doc), ARCHITECTURE, AGENT_GUIDE, SCHEMA reference

**v2+ (hypothetical)**: agent framework developers, technical game designers who prefer declarative data over procedural code, and AI-coding-tooling authors who want an engine that is LLM-native by construction.

## 5. Architecture

### 5.1 System Diagram

```
┌────────────────────────────────────────────────────┐
│  Agent (LLM, e.g. Claude / Codex)                  │
│  - Writes scene JSON, behavior rules, resource specs│
│  - Queries runtime state, runs scenarios, replays  │
└──────────────────────┬─────────────────────────────┘
                       │ TypeScript SDK (typed client)
┌──────────────────────▼─────────────────────────────┐
│  JSON-RPC bridge (Rust, in-process)                │
│  - Typed API surface                               │
│  - Rust types → JSON Schema                        │
│  - NAPI binding for TS                             │
└──────────────────────┬─────────────────────────────┘
┌──────────────────────▼─────────────────────────────┐
│  Engine kernel (Rust)                              │
│  ├─ Scene graph: typed node tree, parent/child     │
│  ├─ Signals: typed event bus, serializable subs    │
│  ├─ Resources: strongly-typed, addressable         │
│  ├─ Behavior runtime: declarative rules + systems  │
│  ├─ Determinism: seeds, tick, replay               │
│  └─ Hot reload: file watch + state diff/merge      │
└──────────────────────┬─────────────────────────────┘
┌──────────────────────▼─────────────────────────────┐
│  Reference game: terminal-rendered snake (v1)      │
│  - Terminal renderer (ANSI, in-kernel)             │
│  - Game content = JSON scenes + behavior rules     │
└────────────────────────────────────────────────────┘
```

### 5.2 Key Architectural Decisions

1. **Same process; RPC is API, not transport**: JSON-RPC runs in-process via NAPI. No network overhead in the v1 loop. Sidecar mode deferred.
2. **Schema is a first-class product**: every Rust type emits JSON Schema. TS SDK types are generated. The agent's API view cannot drift from engine reality.
3. **Game content = pure data**: scenes, behaviors, resources are all JSON. The engine interprets; it does not compile. **This is the core of "AI-native."**
4. **Behavior is data, with a Rust escape hatch**: v1 ships three behavior primitives (state machine, tick rule, signal handler) that compose a closed set of actions. When the closed set is insufficient, agents call into Rust-defined `system` functions through a typed JSON contract.

### 5.3 Engine Kernel Module Boundaries

| Module | Responsibility | Does NOT |
|---|---|---|
| `scene-graph` | Node hierarchy, typed node properties | Interpret behavior |
| `signals` | Event publish/subscribe, typed payloads | Implement state machines |
| `resources` | Typed data containers, addressable, immutable | Cache strategy |
| `behavior` | Interpret declarative rules, schedule systems | Render |
| `determinism` | Seed mgmt, tick control, replay | Recording codec |
| `hot-reload` | File watch, state diff, hot-patch | Version control |

Each module is a separate Rust crate. Dependencies are one-way; modules do not import each other's state (interface through traits).

## 6. Core Abstractions

### 6.1 Scene Graph

A node is a typed entity with parent/child relationships, components, and behaviors. All nodes are expressed as JSON.

```json
{
  "id": "player",
  "type": "Player",
  "parent": "world",
  "components": {
    "position": [10, 10],
    "velocity": [0, 0],
    "direction": "right",
    "score": 0
  },
  "behaviors": [ /* see §6.4 */ ]
}
```

Node types have two layers:
- **Base `Node`**: empty container, hierarchy only.
- **Typed nodes** (user-defined, e.g. `Player`): carry schema constraints. Schemas validate at load time; agent gets structured errors on violation.

### 6.2 Signals

A typed event bus for cross-node communication. Subscriptions can be:
- **Internal**: declared as part of a node's behaviors.
- **External**: agent subscribes via RPC for observation.

```json
// Emit
{ "kind": "emit", "signal": "eaten", "args": { "by": "enemy_1" } }

// Node-internal subscription
{ "kind": "on_signal", "signal": "eaten", "actions": [...] }

// Agent subscription (via RPC)
engine.subscribe("eaten", (event) => { ... });
```

### 6.3 Resources

Strongly-typed, immutable, reusable across scenes. Addressed by `res://path`.

```json
{
  "resource": "res://foods/apple.json",
  "type": "Food",
  "data": { "color": "red", "points": 10, "sprite": "res://sprites/apple.asc" }
}
```

Resources are **immutable**: runtime modification does not affect already-loaded instances. This is critical to prevent agent-induced spooky-action bugs. For mutable data, use components (per-node).

### 6.4 Behaviors

The agent's primary authoring surface. v1 ships three primitives plus one escape hatch.

#### (a) State Machine (per node)

```json
{
  "kind": "state_machine",
  "states": ["idle", "moving", "dead"],
  "initial": "idle",
  "transitions": [
    { "from": "idle", "event": "tick", "to": "moving", "guard": "input != none" },
    { "from": "moving", "event": "tick", "to": "idle", "guard": "input == none" },
    { "from": "*", "event": "collide", "to": "dead" }
  ]
}
```

#### (b) Tick Rule (per node)

```json
{
  "kind": "on_tick",
  "actions": [
    "move:player:input.direction",
    "if:at_food:grow:emit:eat"
  ]
}
```

#### (c) Signal Handler (per node)

```json
{
  "kind": "on_signal",
  "signal": "eaten",
  "actions": [
    "set_state:player.dead = true",
    "emit:game_over",
    "destroy:self"
  ]
}
```

#### (d) Escape Hatch: System Call

When the closed action set is insufficient (pathfinding, math, external algorithms), agents invoke a Rust-defined system:

```json
{
  "kind": "call_system",
  "system": "pathfinding::astar",
  "args": { "from": "player.pos", "to": "target.pos" },
  "result_in": "player.next_path"
}
```

Systems are strongly-typed Rust functions, exposed to the agent via auto-generated schema (signature, parameter types, return type).

### 6.5 Action Vocabulary (closed set)

The verbs the agent can use. v1:

- `set_state` / `get_state` — read/write component
- `emit` — fire signal
- `destroy` / `spawn` — node lifecycle
- `if` — conditional
- `move` / `animate` — basic transform
- `log` — debug output
- `call_system` — invoke Rust system

**The vocabulary is closed** = every action the agent writes is enumerable, validatable, executable, and serializable. The engine never "half-executes" an action.

## 7. Data Flow

### 7.1 Three Core Loops

**Authoring loop** (agent writes a scene):
```
agent writes scene.json
   ↓
Engine: parse → schema validate
   ↓
Fail? → structured error (file, line, JSONPath, suggestion)
Pass? → load into scene graph
```

**Tick loop** (every frame):
```
InputBus → tick scheduler
   ↓
process input / fire on_signal / on_tick / state machine transitions
   ↓
execute actions in dependency order
   ↓
update components → queue new signals
   ↓
Renderer reads components (terminal renderer writes ANSI)
```

**Observation loop** (agent watches the game):
- Subscribe to signal: real-time event stream
- Query state: snapshot of current components
- Both via JSON-RPC; TS SDK exposes `engine.subscribe` and `engine.getState`.

### 7.2 Hot Reload

```
file watcher detects scene.json change
   ↓
re-parse + validate
   ↓
diff old vs new scene graph:
  - node property change → update component
  - new node → spawn
  - removed node → destroy (referenced = error & rollback)
  - behavior change → replace
   ↓
no breaking change → seamless continuation
```

**Invariant**: node IDs are stable. Hot reload preserves them. Agent subscriptions survive.

### 7.3 Replay

Recording = `scene.json + seed + input_log + tick_count`. Replay = re-run with same seed, same input. Any tick can be inspected, paused, stepped. Agents use this for **counterfactual reasoning**: "what would have happened if state machine X had a different transition?"

### 7.4 Invariants

| Invariant | Reason |
|---|---|
| Tick is atomic: an action cannot observe its own emit's effect | Prevent reentrancy; enable replay |
| Node A cannot directly `set_state` node B's component | Force explicit dependencies via signals |
| Hot reload preserves IDs | Agent subscriptions survive |
| System calls are sync-blocking | Async systems would break replay |
| Closed-set actions cannot perform external IO | Behaviors stay replayable and inspectable |

## 8. API Surface (TypeScript SDK)

The SDK is the agent's window into the engine. **Schema is the source of truth**: every Rust type emits JSON Schema; TS types are generated. The agent's view of the API cannot drift from the engine.

### 8.1 Method Surface

```typescript
// ── Engine lifecycle ──
engine.start(scenePath: string): Promise<SceneId>
engine.stop(): Promise<void>
engine.reload(): Promise<void>          // trigger hot reload

// ── Scene / nodes ──
engine.listNodes(): Promise<NodeInfo[]>
engine.getNode(id: NodeId): Promise<NodeState>
engine.getSceneGraph(): Promise<SceneGraph>
engine.validateScene(scene: Scene): Promise<ValidationResult>

// ── Components ──
engine.getComponent<T>(nodeId: NodeId, key: string): Promise<T>
engine.setComponent(nodeId: NodeId, key: string, value: any): Promise<void>

// ── Signals ──
engine.subscribe(signal: string, handler: (event) => void): Unsubscribe
engine.emit(signal: string, args: any): Promise<void>

// ── Resources ──
engine.listResources(): Promise<ResourceInfo[]>
engine.getResource(path: string): Promise<Resource>
engine.registerResource(path: string, data: any): Promise<void>

// ── Replay / debugging ──
engine.startRecording(path: string): Promise<void>
engine.replay(recording: Recording, opts?: { fromTick?: number }): Promise<void>
engine.step(): Promise<void>     // single step
engine.pause(): Promise<void>
engine.resume(): Promise<void>
engine.getStateAtTick(tick: number): Promise<Snapshot>

// ── Schema introspection ──
engine.getSchema(): Promise<JSONSchema>            // all node types, actions, systems
engine.getNodeTypeSchema(type: string): Promise<JSONSchema>
```

### 8.2 Typical Agent Round Trip

```typescript
await engine.start("res://games/snake/scene.json");

// 1) Observe
const playerPos = await engine.getComponent("player", "position");
engine.subscribe("eaten", e => console.log("score:", e.score));

// 2) Modify (write to file, trigger hot reload)
await fs.writeFile(scenePath, updatedScene);
await engine.reload();   // diff & merge

// 3) Counterfactual reasoning
await engine.replay(recording, { fromTick: 42 });
```

### 8.3 Self-Description

`engine.getSchema()` returns the entire engine schema — every node type, every action, every system. The agent queries this on startup. **No hardcoded knowledge of the engine is required.** This is critical: future engine versions require no agent retraining.

## 9. Error Handling

### 9.1 Core Principle

**Errors are first-class structured data.** Agents can reason about them programmatically; the format is consistent across all engine subsystems.

### 9.2 Error Format

```json
{
  "category": "validation",
  "code": "E_SCHEMA_MISMATCH",
  "message": "Node 'player' missing required component 'position'",
  "location": { "file": "scene.json", "line": 42, "path": "$.root.children[0]" },
  "context": { "node_id": "player", "node_type": "Player", "missing_field": "position" },
  "suggestion": "Add 'position' component to node 'player'",
  "severity": "error"
}
```

| Field | Purpose |
|---|---|
| `category` + `code` | Programmatic classification (retry / ignore / abort) |
| `location` | Jump to file position (file + JSONPath) |
| `context` | Surrounding state for diagnosis |
| `suggestion` | Entry point for agent's auto-patch |
| `severity` | `error` / `warning` / `info` |

### 9.3 Error Categories

| Category | When | Example |
|---|---|---|
| `validation` | Load / validate scene | Schema mismatch, ID collision, null reference |
| `hot_reload` | File change diff | Breaking change, rollback failure |
| `runtime` | During tick | Reference to missing node, guard never true |
| `system` | Rust system call | Business error (see system's schema) |
| `replay` | During replay | Tick state divergence (fatal) |

### 9.4 Invariants

- **Engine never panics.** Every panic is converted to a structured error.
- **Validation runs once at load.** Schema errors do not surface during ticks.
- **Runtime errors are warnings, not fatal.** A single action's failure does not abort the tick (unless the system marks it fatal).
- **Replay divergence is fatal.** State mismatch indicates a bug; stop immediately.
- **Suggestions are schema-derived**, not LLM-fabricated. They are structured, executable patch hints.

## 10. Testing Strategy

### 10.1 Engine Kernel

- **Unit tests**: per-module (scene graph, signals, resources, behavior, determinism).
- **Property tests** (`proptest`): "any scene loads + serializes → byte-equal"; "any schema change → JSON Schema remains valid."
- **Replay regression**: reference game recordings (1000+ ticks) checked in. Every change re-runs replay + hash check.

### 10.2 AI Agent Evaluation (Core, Novel)

Three layers:

| Layer | Question | Method |
|---|---|---|
| Compile | Does the agent's scene load? | Engine validation pass rate |
| Behavior | Does the game run? | Recording + deterministic replay + hash equality |
| Task | Does the agent complete the assigned task? | Benchmark task suite: snake, puzzle, tower defense, dialog tree. Each has explicit pass criteria. |

The benchmark task suite is the engine's "exam." Agent takes the task, writes scenes, engine judges. Can run manually or in CI (cost permitting).

### 10.3 Reference Game's Dual Role

Not just a demo — also the integration test. Any engine change must keep the reference game working:
- Load + run 1000 ticks + replay + hash check = integration test passes.

### 10.4 Testable Invariants

| Invariant | Test |
|---|---|
| Tick atomicity | Unit: emit in same tick does not change state |
| Hot reload preserves IDs | Unit: subscription still fires after reload |
| Replay reproducibility | Integration: recording + replay hash equal |
| Schema roundtrip | Property: scene JSON byte-equal after parse/serialize |
| Closed actions don't perform IO | Static analysis + runtime check |

## 13. Open Questions

1. **Sandboxed agent execution**: v1 has no sandbox. Can the agent crash the engine? Should we add a "safe mode" where the engine only runs in observation mode? *Defer to v2 unless a clear need emerges during M1–M5.*
2. **Multiplayer / network replay**: out of scope for v1, but the determinism layer is a precondition. Should we document a future path? *Note in ARCHITECTURE.md, not block v1.*
3. **License of the reference game art (ASCII)**: trivial. *MIT, same as engine.*
4. **What does an "agent" actually look like in the loop?** A CLI that calls the SDK? A long-running process? An LLM in a chat session? *Decide during M6-M7. SDK should be ergonomic from a Node process; LLM agent layer is a thin wrapper.*
5. **Performance budgets**: how many nodes / signals / actions per tick are we targeting? *Set during M3, validate with reference game.*

## 14. Glossary

- **Agent** — an LLM-driven system that uses the Craft SDK to author or modify game content.
- **Action** — a single closed-set verb the engine understands (e.g. `set_state`, `emit`).
- **Behavior** — a declarative rule attached to a node (state machine / tick rule / signal handler).
- **Component** — typed, per-node mutable data. The agent reads/writes these.
- **Resource** — typed, immutable, reusable data. The agent creates these but does not mutate at runtime.
- **Scene graph** — the tree of nodes that constitutes a game's object hierarchy.
- **Signal** — a typed event published on the engine's event bus.
- **System** — a Rust function with a typed JSON contract, callable from behavior. The escape hatch.
- **Tick** — one unit of game simulation. Atomic from the agent's perspective.
- **Replay** — deterministic re-execution of a recorded (scene + seed + input) sequence.
